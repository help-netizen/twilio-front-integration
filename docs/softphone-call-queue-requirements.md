# Функциональные требования — очередь звонков SoftPhone для нескольких диспетчеров

## Статус

- Draft

## Контекст

Tenant-компания может иметь нескольких пользователей-диспетчеров, работающих одновременно.
В проекте уже существует визуальный индикатор SoftPhone в верхней навигации. Для свободного диспетчера он используется как основной входящий индикатор звонка.

Проблема, которую нужно решить:

- если диспетчер уже разговаривает, новый входящий звонок не должен вмешиваться в текущий разговор;
- занятый диспетчер не должен слышать рингтон второго звонка;
- при этом занятый диспетчер должен видеть, что в очереди tenant-компании есть ожидающий звонок;
- свободные диспетчеры должны продолжать получать текущий сценарий входящего звонка: рингтон и активный бейдж/индикатор в навигации.

Цель документа: зафиксировать требования к очереди входящих вызовов на стороне SoftPhone без изменения текущего паттерна навигационного индикатора.

## Область действия

Документ покрывает:

- поведение общей очереди входящих вызовов tenant-компании;
- поведение SoftPhone UI для свободных и занятых диспетчеров;
- правила показа рингтона и бейджа в навигации;
- синхронизацию состояния между несколькими диспетчерами одной tenant-компании.

Документ не покрывает:

- redesign всего SoftPhone;
- call hold / swap / conference;
- логику тарификации, записи звонков и пост-обработки;
- изменения IVR, voicemail и внешнего call-routing за пределами описанной очереди.

## Термины

- `Tenant-компания` — изолированная компания в системе.
- `Диспетчер` — пользователь tenant-компании с доступом к телефонии.
- `Свободный диспетчер` — диспетчер без активного разговора в SoftPhone.
- `Занятый диспетчер` — диспетчер, у которого уже есть активный разговор.
- `Общая очередь компании` — набор входящих вызовов tenant-компании, которые еще не были приняты финально.
- `Ожидающий вызов` — входящий вызов, который еще не принят ни одним диспетчером и не завершен финальным статусом.
- `Навигационный бейдж SoftPhone` — существующий индикатор/кнопка SoftPhone в верхней навигации.

## Базовые принципы

1. Входящий вызов принадлежит tenant-компании, а не одному UI-экземпляру SoftPhone.
2. Один и тот же ожидающий вызов может быть видим нескольким диспетчерам tenant-компании до момента принятия одним из них.
3. Поведение уведомлений должно зависеть от занятости конкретного диспетчера.
4. Новый ожидающий вызов не должен ломать интерфейс и управление уже активным разговором.
5. Существующий навигационный индикатор SoftPhone должен оставаться главным элементом уведомления, а не заменяться отдельным новым глобальным виджетом.

## Функциональные требования

### 1. Общая очередь tenant-компании

1. Система должна вести общую очередь ожидающих входящих вызовов отдельно для каждой tenant-компании.
2. В очередь tenant-компании должны попадать все входящие вызовы, которые еще не были приняты, отклонены вызывающим, переведены в voicemail или завершены иным финальным статусом.
3. Очередь tenant-компании должна быть изолирована от других tenant-компаний.
4. Каждый ожидающий вызов должен иметь уникальный идентификатор, позволяющий синхронно обновлять его состояние на всех клиентах диспетчеров.
5. Очередь tenant-компании должна поддерживать минимум один активный ожидающий вызов и произвольное количество дополнительных ожидающих вызовов.
6. Порядок вызовов в очереди должен быть детерминированным.
7. Базовый порядок очереди должен быть FIFO по времени поступления, если иное правило не задано отдельно.

### 2. Участники распределения

1. В обработке вызовов tenant-компании должны участвовать только диспетчеры этой tenant-компании.
2. Получать события SoftPhone-вызовов должны только пользователи tenant-компании, у которых есть право на телефонию.
3. Для каждого диспетчера система должна отдельно определять его текущее состояние как `свободен` или `занят`.
4. Статус занятости должен определяться по наличию активного разговора у конкретного диспетчера, а не по наличию вызовов в очереди компании.

### 3. Поведение для свободного диспетчера

1. Если диспетчер свободен и в tenant-компанию поступает новый входящий вызов, диспетчер должен получить текущее активное уведомление о входящем звонке.
2. Активное уведомление для свободного диспетчера должно включать рингтон.
3. Активное уведомление для свободного диспетчера должно включать существующий активный индикатор входящего вызова на кнопке SoftPhone в навигации.
4. Для свободного диспетчера внешний вид уведомления должен оставаться максимально близким к текущей реализации.
5. Свободный диспетчер должен иметь возможность принять вызов стандартным способом через текущий UI SoftPhone.

### 4. Поведение для занятого диспетчера

1. Если диспетчер уже находится на линии, новый входящий вызов tenant-компании не должен запускать у него рингтон.
2. Если диспетчер уже находится на линии, новый входящий вызов tenant-компании не должен переводить его SoftPhone в состояние, перехватывающее текущий разговор.
3. Если диспетчер уже находится на линии, новый входящий вызов tenant-компании не должен подменять экран активного разговора.
4. Если диспетчер уже находится на линии, новый входящий вызов tenant-компании не должен сбрасывать таймер, mute-состояние и элементы управления текущего разговора.
5. Если диспетчер уже находится на линии, он должен видеть только тихий визуальный индикатор ожидания на существующем навигационном элементе SoftPhone.
6. Тихий визуальный индикатор для занятого диспетчера должен показывать, что в очереди tenant-компании есть минимум один ожидающий вызов.
7. Тихий визуальный индикатор для занятого диспетчера не должен требовать немедленного действия и не должен мешать управлению текущим разговором.
8. Новый ожидающий вызов не должен иметь возможности остановить, завершить, заглушить или иным образом изменить текущий разговор занятого диспетчера.

### 5. Требования к бейджу в навигации

1. Навигационный бейдж SoftPhone должен оставаться единой точкой индикации телефонного состояния пользователя.
2. Для свободного диспетчера при входящем вызове бейдж должен работать по текущей логике активного входящего звонка.
3. Для занятого диспетчера бейдж должен показывать дополнительный признак ожидающего вызова без замены статуса текущего активного звонка.
4. Если у занятого диспетчера есть активный разговор и одновременно в очереди tenant-компании есть ожидающие вызовы, UI бейджа должен в первую очередь сохранять контекст текущего разговора.
5. При наличии ожидающих вызовов бейдж должен дополнительно показывать индикатор ожидания в каком-либо компактном виде.
6. Если ожидающих вызовов больше одного, бейдж должен показывать количество ожидающих вызовов.
7. Если панель SoftPhone открыта, индикатор ожидающих вызовов в навигации не должен исчезать.
8. Если ожидающих вызовов нет, дополнительный индикатор очереди не должен отображаться.

### 6. Принятие вызова одним из диспетчеров

1. Первый диспетчер tenant-компании, который принял ожидающий вызов, должен становиться владельцем этого вызова.
2. После успешного принятия вызова одним диспетчером этот вызов должен немедленно исчезать из очереди ожидания у всех остальных диспетчеров tenant-компании.
3. После успешного принятия вызова одним диспетчером рингтон у остальных свободных диспетчеров должен быть немедленно остановлен.
4. После успешного принятия вызова одним диспетчером тихий бейдж ожидания у занятых диспетчеров должен быть немедленно обновлен.
5. Принятие одного вызова не должно затрагивать другие еще живые вызовы в очереди tenant-компании.

### 7. Завершение и очистка ожидающих вызовов

1. Если ожидающий вызов был отменен вызывающим, он должен исчезнуть из очереди tenant-компании у всех диспетчеров.
2. Если ожидающий вызов перешел в voicemail или другой финальный статус, он должен исчезнуть из очереди tenant-компании у всех диспетчеров.
3. Если ожидающий вызов исчез из очереди, связанные с ним рингтон и бейджи должны быть обновлены синхронно.
4. Завершение текущего разговора занятого диспетчера не должно автоматически удалять живой ожидающий вызов из очереди tenant-компании.

### 8. Поведение при освобождении диспетчера

1. Если занятый диспетчер завершил свой текущий разговор, система должна немедленно пересчитать его статус как `свободен`.
2. Если после этого в очереди tenant-компании еще есть живой ожидающий вызов, диспетчер должен начать получать стандартный сценарий входящего вызова.
3. Стандартный сценарий после освобождения диспетчера должен включать рингтон и активный входящий индикатор в навигации.
4. При наличии нескольких живых ожидающих вызовов после освобождения диспетчера должен выбираться следующий вызов согласно порядку очереди.

### 9. Real-time синхронизация

1. Все изменения очереди tenant-компании должны синхронизироваться в реальном времени между всеми диспетчерами этой tenant-компании.
2. События должны распространяться как минимум для следующих переходов: новый входящий вызов, принятие вызова, отмена вызывающим, финализация вызова, изменение размера очереди.
3. UI всех диспетчеров tenant-компании должен сходиться к одному и тому же состоянию очереди без ручного обновления страницы.
4. Сбой обновления у одного клиента не должен менять фактическое владение вызовом на стороне других клиентов.

### 10. Многосессионность одного диспетчера

1. Если один и тот же диспетчер авторизован в нескольких браузерных сессиях или вкладках, состояние очереди и бейджа должно быть консистентным между его сессиями.
2. Принятие вызова в одной сессии должно немедленно убирать этот вызов из других сессий того же диспетчера.
3. Рингтон не должен продолжать звучать в другой сессии после того, как вызов уже принят в первой.

### 11. Ограничения текущего релиза

1. В рамках этой задачи не требуется внедрять hold текущего разговора.
2. В рамках этой задачи не требуется внедрять swap между разговорами.
3. В рамках этой задачи не требуется показывать занятому диспетчеру кнопки `Accept` или `Decline` для второго вызова поверх активного разговора.
4. В рамках этой задачи ожидающий вызов для занятого диспетчера должен быть purely informational через бейдж в навигации.

## Нефункциональные требования

1. Появление ожидающего вызова не должно ухудшать управляемость текущего разговора.
2. Обновление бейджа и очереди должно происходить без полной перезагрузки страницы.
3. UI-индикация должна быть визуально совместима с текущим стилем SoftPhone в навигации.
4. Система не должна допускать состояния, в котором занятый диспетчер слышит рингтон, но не видит соответствующего индикатора ожидающего вызова.
5. Система не должна допускать состояния, в котором вызов уже принят одним диспетчером, но продолжает отображаться как доступный у других.

## Критерии приемки

1. Если в tenant-компании 3 диспетчера, из которых 2 свободны и 1 занят, при новом входящем вызове оба свободных диспетчера должны получить рингтон и активный бейдж входящего вызова, а занятый диспетчер должен увидеть только тихий бейдж ожидания без рингтона.
2. Если один из свободных диспетчеров принимает вызов, у второго свободного диспетчера должен остановиться рингтон и исчезнуть активный входящий индикатор, а у занятого должен обновиться или исчезнуть тихий бейдж по этому вызову.
3. Если в tenant-компании все диспетчеры заняты, новый входящий вызов не должен запускать рингтон ни у одного из них, но у всех должен появиться тихий бейдж ожидания.
4. Если при общей занятости один диспетчер завершает свой разговор, а ожидающий вызов еще жив, этот диспетчер должен начать получать стандартный сценарий входящего вызова с рингтоном и активным бейджем.
5. Во время активного разговора у занятого диспетчера таймер, mute и все элементы управления текущим звонком должны сохраняться без изменений при появлении ожидающего вызова.
6. Если у tenant-компании в очереди два и более живых вызова, навигационный бейдж должен корректно показывать наличие очереди и актуальное количество ожидающих вызовов.
7. Если ожидающий вызов отменен вызывающим до ответа, он должен исчезнуть из бейджа и очереди у всех диспетчеров tenant-компании.
8. Если один и тот же диспетчер открыт в двух вкладках, принятие вызова в одной вкладке должно немедленно остановить рингтон и убрать этот вызов из второй вкладки.

## Ожидаемый результат

После внедрения требований SoftPhone должен поддерживать безопасное и предсказуемое поведение очереди входящих вызовов для нескольких диспетчеров одной tenant-компании:

- свободные диспетчеры получают активное уведомление и могут принять вызов;
- занятые диспетчеры не теряют текущий разговор и видят только тихую индикацию ожидания;
- очередь компании остается общей и синхронной для всех участников;
- навигационный индикатор SoftPhone продолжает быть основным и знакомым пользователю UI-элементом.
