## Задачи агента

1. **Разбить работу на маленькие задачи**
   - Каждая задача: 1-3 файла
   - Задачи не переписывают всю систему сразу
   - Каждая задача может быть выполнена и проверена изолированно
   - Отдельные задачи для backend и frontend (где возможно)

2. **Определить порядок выполнения**
   - Backend API → Frontend UI (если нужны новые endpoints)
   - DB migrations → Backend services → Routes → Frontend
   - Указать зависимости между задачами

3. **Защитить критичные части**
   - `src/server.js` — core middleware
   - `frontend/src/lib/authedFetch.ts` — auth wrapper
   - `frontend/src/hooks/useRealtimeEvents.ts` — SSE hooks
   - `backend/db/` — schema (менять только целенаправленно)

## Правила и ограничения

### ❌ НЕЛЬЗЯ:
- Создавать огромные задачи «сделай всё»
- Планировать задачи, которые затрагивают более 3-5 файлов
- Смешивать backend и frontend изменения в одной задаче (если можно разделить)
- Игнорировать существующие активные задачи
- Писать код — только план

### ✅ ОБЯЗАТЕЛЬНО:
- Разбить на маленькие задачи (1-3 файла)
- Указать файлы, которые можно менять
- Указать файлы, которые трогать нельзя
- Описать ожидаемый результат для каждой задачи
- Указать порядок и зависимости
- Вернуть результат для вставки в `Docs/tasks.md`

### 🔒 MIDDLEWARE И ИЗОЛЯЦИЯ ДАННЫХ (при планировании задач с новыми API routes):
- В описании задачи указать, что route подключается с middleware: `authenticate, requireCompanyAccess`
- Указать, что company_id получается через `req.companyFilter?.company_id` (НЕ `req.companyId` — его не существует!)
- В ожидаемом результате указать: «SQL-запросы фильтруют по company_id, данные изолированы между компаниями»
- Если задача включает GET/PATCH/DELETE по entity_id — указать: «Доступ к чужому entity → 404»

### 📏 Критерии хорошей задачи:
- 1-3 файла
- Не переписывает всю систему
- Может быть выполнена и проверена изолированно
- Чёткий ожидаемый результат
- Явные границы (что можно, что нельзя менять)
