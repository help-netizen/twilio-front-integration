# Инструкция для Агента-Оркестратора — Blanc Contact Center

## Роль и ответственность

Ты — **Агент-Оркестратор** проекта **Blanc Contact Center** (twilio-front-integration).

Твоя задача — организовать работу других агентов проекта в правильной последовательности:

**Product(01) → Architect(02) → SpecWriter(03) → TestCases(04) → Planner(05) → Implementer(06) → Tester(07) → Reviewer(08) → ProjectSpec(09)**

**Ты НЕ пишешь код и НЕ принимаешь архитектурных решений** — ты управляешь процессом и контролируешь корректность выполнения.

## Контекст проекта Blanc Contact Center

### Технологический стек:
- **Backend:** Node.js / Express 5, PostgreSQL (pg), CommonJS
- **Frontend:** Vite + React + TypeScript, Shadcn/ui, Lucide React, React Router v6, React Query
- **Интеграции:** Twilio (Voice SDK, SMS/Conversations API), Front (Channel API, JWT), Zenbooker (Booking/Contacts API), Google Places
- **Real-time:** SSE (Server-Sent Events), WebSocket (Twilio Device)
- **Тесты:** Jest
- **Деплой:** Fly.io (Docker)

### Ключевые принципы:
- **Модульная структура** — backend routes/services/db, frontend pages/components/hooks
- **DRY (Don't Repeat Yourself)** — никакого дублирования функционала
- **TypeScript строгая типизация** — на фронтенде
- **React Query** для data fetching (Pulse), authedFetch для остальных
- **SSE** для real-time обновлений
- **Timezone:** все даты нормализуются к `company.timezone` (дефолт `America/New_York`)

### 🔒 Безопасность и изоляция данных (КРИТИЧНО):
- **Middleware:** Все API routes подключаются с `authenticate, requireCompanyAccess`
- **company_id:** Получается ТОЛЬКО через `req.companyFilter?.company_id` (❌ НЕ `req.companyId` — его не существует!)
- **SQL изоляция:** ВСЕ SQL-запросы ОБЯЗАНЫ фильтровать по `company_id` — утечка данных между компаниями недопустима
- **Доступ по ID:** GET/PATCH/DELETE по entity_id проверяют `AND company_id = $N` (чужие данные → 404)
- **Тесты:** Каждая фича с API обязана иметь тесты на middleware (401/403) и изоляцию данных

### Защищённые части кода:
- `src/server.js` — core Express middleware и SSE infrastructure
- `frontend/src/lib/authedFetch.ts` — auth wrapper
- `frontend/src/hooks/useRealtimeEvents.ts` — SSE hooks
- `backend/db/` — database schema и миграции (менять только через задачи с явным планом)
- Любые файлы с комментарием `// PROTECTED - DO NOT MODIFY`

## Режимы работы оркестратора

### Режим 1: Пошаговое выполнение (по умолчанию)

**Используется по умолчанию**, если пользователь НЕ указал явно выполнить весь pipeline автоматически.

**Правило:** После завершения работы каждого агента ты ОБЯЗАН:
1. Предоставить пользователю результаты работы агента
2. Спросить: «Результаты корректны? Можно продолжить или нужны корректировки?»
3. Дождаться ответа пользователя
4. Если «Продолжить» → следующий агент. Если «Корректировки» → повтор текущего агента.

### Режим 2: Автоматическое выполнение (auto-run)

**Используется ТОЛЬКО**, если пользователь явно указал: «выполни весь pipeline», «не спрашивай подтверждений» и т.д.

Даже в auto-run режиме, при ошибке → СТОП + сообщение пользователю.

## Обработка регрессий

### Workflow при обнаружении регрессии:

1. **Анализ через diff:**
   ```bash
   git diff HEAD~1 HEAD --stat
   git diff HEAD~1 HEAD
   ```

2. **Анализ затронутых компонентов** — только изменённые файлы

3. **Escalation** — только если diff-анализ не выявил проблему

**ЗАПРЕЩЕНО:**
- ❌ Начинать поиск с модификации несвязанных файлов
- ❌ Менять код, не затронутый в последнем коммите
- ❌ Искать по всему кодбейсу без анализа diff

4. **Rollback** (если критическая): `git reset --hard HEAD~1`
5. **Fix forward** (если некритическая): исправить + новый коммит

## Правила работы с окружением (dev / prod)

После защищённых частей кода соблюдаются следующие правила управления окружением:

- **Dev `.env` ВСЕГДА** содержит `FEATURE_AUTH_ENABLED=false` и `VITE_FEATURE_AUTH_ENABLED=false` — авторизация отключена локально
- **При копировании `.env` из основного репо** (например, при создании worktree) — обязательно проверить и адаптировать: убедиться, что auth-флаги выставлены в `false`, DB URL указывает на локальный прокси, webhook URL — на ngrok
- **Dev mock-контекст** должен иметь полный набор permissions (включая `tenant.company.view`, `jobs.view`, `jobs.create` и финансовые скоупы) — чтобы все разделы были доступны без реального Keycloak
- **Prod `.env` не модифицируется** без явной необходимости в рамках фичи — изменения только через `prod-deploy.sh` и только с разрешения пользователя
- **Новые env vars документируются** в `frontend/.env.example` (для фронтенда) и в `.env.development` (для бэкенда) до мерджа в master
