# MOBILE-TECH-APP-001 — нативное iOS-приложение для техников (дизайн)

**Статус:** Design / pre-build · **Дата:** 2026-07-02 · **Продукт:** Albusto (в UI — только «Albusto»)
**Тип:** greenfield нативный клиент + минимальные добавления на существующий backend. Кода ещё НЕТ — это проектирование «как должно быть» перед разработкой.

---

## 1. Цель и границы

**Цель v1:** нативное iOS-приложение для полевого техника (роль `provider`), которое:
- показывает его расписание и работы, **работает офлайн на чтение** (данные с последней синхронизации живут на устройстве);
- позволяет вести работу: сменить статус, добавить заметку/фото, проложить маршрут, позвонить;
- **принимает оплату картой на месте через Tap to Pay on iPhone** (NFC, без доп-железа);
- шлёт нативные пуши (новое назначение / изменение расписания).

**Не-цели v1 (осознанно отложено):**
- **In-app VoIP-звонки (Twilio Voice + CallKit)** → **v2** (в v1 звонок = системный набор `tel:`).
- Полный offline-first с офлайн-записью и разрешением конфликтов → возможно v2, если read-only начнёт мешать.
- Android → позже (v1 = только iOS, по решению владельца).
- Роли кроме `provider` (диспетчер/админ остаются в веб-приложении).

## 3. Технологический стек

- **Клиент:** React Native (**Expo, bare/dev-client workflow** — рекоменд.: Expo prebuild + config-plugins даёт нативные модули CallKit/Stripe Terminal/APNs, при этом сохраняет удобство RN и переиспользование TS/логики веба). Язык — TypeScript (общий с вашим `frontend/`).
- **Локальная БД:** SQLite (через `op-sqlite`/`expo-sqlite`); read-only кэш (см. §5).
- **Сеть/кэш:** переиспользуем контракты API из веба; TanStack Query + persist-адаптер поверх SQLite (или тонкий свой слой) для «показать из кэша → обновить при сети».
- **Авторизация:** OAuth2 **PKCE (S256)** против Keycloak realm `crm-prod` через `react-native-app-auth`; токены в **iOS Keychain**.
- **Платежи:** `@stripe/stripe-terminal-react-native` → **Tap to Pay on iPhone**, поверх существующего Stripe (Connect/PaymentIntents).
- **Пуши:** APNs (`expo-notifications` / нативно); VoIP-пуши (PushKit) — только в v2 под CallKit.
- **Карты/звонок (v1):** deep-link в Apple Maps (`maps://` / universal link) + `tel:`.

**Переиспользование с веба:** TypeScript-типы сущностей (job/schedule/note), API-клиентские контракты, доменная логика статусов (FSM `blanc_status`), дизайн-токены Albusto. UI — нативный (не webview).

## 4. Архитектура клиента (слои)

```
┌──────────────────────────────────────────────┐
│ UI (RN, нативные экраны, дизайн Albusto)       │  Расписание · Работа · Оплата · Настройки
├──────────────────────────────────────────────┤
│ Феатур-логика / состояние (TanStack Query)     │  «из кэша → обновить»; онлайн-гейт на записи
├──────────────────────────────────────────────┤
│ Sync-движок (read-only, §5)                    │  delta-pull по last_synced_at → upsert в SQLite
├──────────────┬───────────────┬────────────────┤
│ SQLite кэш    │ API-клиент     │ Auth (Keychain)│
│ (jobs, notes, │ authedFetch    │ Keycloak PKCE  │
│  schedule)    │ + токен        │ refresh/offline│
└──────────────┴───────────────┴────────────────┘
        ↑ нативные модули: Camera · Location · Maps · StripeTerminal(TapToPay) · APNs
```

- **Источник правды для чтения** — SQLite. Экран рисуется мгновенно из кэша, затем sync-движок тихо обновляет. Нет сети → рисуем кэш + баннер «офлайн, данные от {время последней синхронизации}».
- **Записи (статус/заметка/фото/оплата)** идут прямо на backend, **только при сети**. Нет сети → кнопка неактивна с явным «нет соединения» (осознанный выбор read-only; без очереди/конфликтов).

## 5. Офлайн-модель (read-only) и синхронизация

**Что кэшируем на устройстве (провайдер-scoped):** назначенные работы (окно ~2 недели + все открытые), их детали (клиент, адрес, время, статус, заметки, история), метаданные фото. Объём: ~50–100 МБ SQLite + локальный кэш фото — тривиально (у техника ~10–50 работ/нед).

**Стратегия sync (простая, под read-only):**
1. Клиент хранит `last_synced_at`.
2. При старте/фореграунде/pull-to-refresh → **delta-pull**: «мои назначенные работы + заметки, изменённые с `last_synced_at`».
3. Upsert в SQLite по `id`; удаления — по тумбстоунам (см. бэкенд §8).
4. `last_synced_at = max(updated_at)` из ответа.

**Почему нужен новый delta-endpoint (§8):** сейчас списки — `offset/limit` полным снимком, без `?since=`/курсора/ETag (подтверждено разведкой). Для лёгкого офлайна добавляем **read-only delta-endpoint**, провайдер-scoped. У всех таблиц уже есть `updated_at` — добавление дешёвое. Конфликтов/мерджа нет (read-only) — это главное упрощение.

## 6. Авторизация (нативно)

- **Флоу:** `react-native-app-auth` → PKCE S256 → Keycloak realm `crm-prod`, **отдельный public-клиент `crm-mobile`** (новый; custom-scheme redirect `albusto://auth`). Веб-клиент `crm-web` не трогаем.
- **Хранение:** access/refresh — в Keychain; refresh делает сам SDK против token-endpoint Keycloak (наш backend в рефреше не участвует).
- **Офлайн-сессия:** для read-only офлайна живой токен не нужен — отдаём кэш; на первом же онлайне — silent refresh. Кэшируем `/api/auth/me` контекст (company_id, crm_user_id, role, scopes) — **критично** для провайдер-скоупа (без `crm_users.id` видимость деградирует в ноль — deny-by-default).
- **2FA/сессии:** наследуем существующую политику (30-дневные сессии); биометрика (FaceID) — локальный анлок закэшированной сессии.

## 7. Фичи v1 (по каждой: API + онлайн/офлайн)

| Фича | Экран | API (в основном существует) | Офлайн |
|---|---|---|---|
| **Расписание** (день/неделя) | Список по дням | `GET /api/schedule` (провайдер-scoped, jobs) | ✅ чтение из кэша |
| **Детали работы** | Карточка | `GET /api/jobs/:id` + `/notes` | ✅ чтение из кэша |
| **Смена статуса** | Кнопки на карточке | `POST /api/jobs/:id/{enroute,start,complete}` (FSM; у провайдера есть права на свои работы) | ❌ только онлайн |
| **Заметки + фото** | В карточке | `POST /api/jobs/:id/notes` (multipart, фото с камеры) | ❌ только онлайн |
| **Навигация** | Кнопка «В путь» | deep-link Apple Maps по адресу работы | ✅ (адрес из кэша) |
| **Звонок клиенту** | Кнопка | `tel:` системный набор (v1) | ✅ (номер из кэша) |
| **Приём оплаты** | Экран «Оплата» | **Stripe Terminal Tap to Pay** → connection token + PaymentIntent (§8) | ❌ только онлайн (платёж требует сети) |
| **Пуши** | — | APNs: регистрация device-token + отправка при назначении/переносе (§8) | — |

**Оплата (Tap to Pay) — поток:** открыть работу → «Принять оплату» → сумма → `discoverReaders` (localMobile) → `collectPaymentMethod` (клиент прикладывает карту к iPhone) → `confirmPaymentIntent` → чек/статус в CRM. Онбординг: Stripe Terminal на аккаунте + Apple-entitlement (§10).

## 8. Изменения на backend (минимальные, аддитивные)

1. **Delta-sync endpoint (read-only, провайдер-scoped):** `GET /api/sync/jobs?since=<ts>` → изменённые/удалённые (тумбстоуны) назначенные работы + их заметки. Переиспользует `getProviderScope` (assigned_only) и существующие `updated_at`. Курсор — по `updated_at`+`id` (стабильный, как `getUnifiedTimelinePage`).
2. **Keycloak mobile-клиент `crm-mobile`:** public, PKCE, redirect `albusto://auth` (realm-export + setup-скрипт, как для google-idp).
3. **APNs push:** таблица `device_tokens (crm_user_id, apns_token, platform)`; отправка при назначении провайдера на работу / переносе (хук в существующий reassign/schedule путь). SSE у нас browser-only — для натива нужен APNs.
4. **Stripe Terminal:** `POST /api/payments/terminal/connection-token` + создание/подтверждение PaymentIntent для Tap to Pay (расширение существующей Stripe-интеграции — Connect/payment sessions уже есть).
5. **(v2)** Twilio Voice: `/api/voice/token` уже отдаёт AccessToken+VoiceGrant (1 час) — для CallKit добавить refresh + VoIP-push (PushKit) маршрут.

Всё — с обязательными `authenticate`+`requireCompanyAccess`, провайдер-скоуп, company_id только из токена, тесты 401/403+изоляция.

## 9. Фазы

- **v1 (ядро):** RN-приложение · auth · офлайн-read кэш + delta-sync · расписание/детали · статусы · заметки+фото · навигация · `tel:` · **Tap to Pay** · APNs-пуши.
- **v2:** In-app VoIP (Twilio Voice iOS SDK + **CallKit** + VoIP-push) — заменяет `tel:`; фоновые входящие как системные звонки.
- **v3+ (если понадобится):** офлайн-записи с безопасной очередью (append-only заметки/статусы) + разрешение конфликтов; Android; фоновая геолокация/геозоны (авто «на месте»).

## 10. Длинные лид-таймы и риски (начать заранее)

- ⚑ **Tap to Pay entitlement** `com.apple.developer.proximity-reader.payment.acceptance` — запрашивается у Apple (аппрув ~дни) + Stripe Terminal онбординг. Начать до кода оплаты.
- ⚑ **VoIP-push entitlement (v2)** — под CallKit.
- **Токены Keycloak коротко-живущие** — refresh в фоне; офлайн read-only это обходит (отдаём кэш).
- **Провайдер-скоуп deny-by-default** — если кэш профиля рассинхронён (`crm_users.id`), видимость = ноль. Кэшировать контекст на логине, ревалидировать на старте.
- **Blanc-status vs Zenbooker** — статус может разойтись с ZB-синком; показывать оба, предупреждать при расхождении на реконнекте.
- **Caller ID при v2-VoIP** — исходящие с номера компании (не личного техника) — уже так в софтфоне; сохранить.
