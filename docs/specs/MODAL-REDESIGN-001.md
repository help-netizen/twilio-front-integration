# MODAL-REDESIGN-001 — Аудит и редизайн модальных окон

> UX/дизайн-документ. Не углубляется в техническую реализацию — описывает реестр,
> классификацию, единые правила, концепцию и рекомендации. Привязан к существующей
> дизайн-системе **Blanc** и к уже имеющимся паттернам (`Dialog`, `FloatingDetailPanel`).

## 1. Краткий вывод

В системе ~44 модальных окна. Все формы рендерятся через один общий компонент
`components/ui/dialog.tsx`, у которого десктопный размер по умолчанию — **`max-w-lg` (512px)**
по центру экрана. Это и есть корень проблемы: многополевые формы ужимаются в маленький
центральный блок, элементы мельчают, появляется внутренний скролл.

При этом в системе **уже есть** правильный широкий формат — `FloatingDetailPanel`
(боковая «плавающая» панель с вариантом `--wide`), который используется для детальных
карточек (Job, Estimate, Lead, Contact …). То есть нужный паттерн существует — его надо
обобщить на формы, а центральный `Dialog` дополнить размерными вариантами.

**Решение (выбранное направление):** формы открываются **боковым «слоем» справа** —
той же механикой, что карточка работы (`FloatingDetailPanel`): панель прижата к правому
краю, на всю высоту, выезжает справа, список под ней остаётся виден. Реализовано как
`variant="panel"` у общего `DialogContent`. Простые подтверждения остаются компактными
по центру (`size`). **Широкие центральные модалки не используем** (по требованию заказчика).

**Канон панели (drawer) — `dialog.tsx`:**
- **Структура `flex`-колонкой:** закреплённая шапка (`DialogPanelHeader`) → прокручиваемое
  тело (`DialogBody`) → липкий футер с действиями (`DialogPanelFooter`). Скроллится **только
  тело**, шапка и действия всегда на месте.
- **Одна колонка.** Поток формы — в один столбец; рядом ставим только короткие тесно
  связанные поля (Имя | Телефон, Город | Штат | Индекс). Никаких 2-колоночных «широких» форм.
- **Размерные тиры ширины** (`PANEL_WIDTH`): `sm` 440 · `default` 520 · `wide` 680 · `full` 820px.
  Обычная форма — `default` (520); документ/сложная — `wide`/`full`.
- **Никаких статичных линий-разделителей.** Вместо `border-top/bottom` у шапки/футера —
  **мягкая тень по краю тела, появляющаяся только при скрытом контенте** (scroll-aware):
  сверху, когда проскроллено; снизу, когда есть что показать ниже. В покое панель чистая.
- **Масштаб как на основных экранах:** поля 36px, лейблы 13px, eyebrow 11px, радиус 10/22px,
  токены Blanc. Не мельчим и не укрупняем — единый масштаб с приложением.
- **Сложные пикеры, которым нужна ширина** (календарь выбора слота `CustomTimeModal`, 1024px),
  открываются **отдельной модалкой по центру поверх панели** — это осознанное исключение:
  широкий календарь не помещается в 520px и заслуживает сфокусированного шага.
- **Мобайл:** та же структура превращается в нижний лист (bottom-sheet) на всю ширину —
  тело скроллится, футер так же липкий снизу.

## 2. Единые правила (rule → формат)

| Тип окна | Признак | Формат | Размер (desktop) |
|---|---|---|---|
| **Подтверждение** | короткий текст + 1–2 кнопки, 0 полей | компактный центр (как есть) | `size="sm"/"default"` |
| **Форма (любая, 2+ поля)** | любая форма создания/редактирования/настройки | **боковой слой справа** (как карточка работы) | `variant="panel"` (≈600px) |
| **Сложный сценарий / документ** | мастер, 10+ полей, Estimate/Invoice | боковой слой справа (шире) / выделенная страница | `variant="panel"` / `FloatingDetailPanel --wide` / page |
| **Длинный read-only контент** | детальная карточка, превью, история | боковая панель `FloatingDetailPanel` | `--wide` |

Правила-инварианты (раздел 8 ТЗ):
- использовать бóльшую часть экрана, минимизировать/убрать внутренний скролл;
- поля/кнопки/типографика — в масштабе основных экранов (не мельче);
- единообразие однотипных сценариев; визуально — часть современного интерфейса, не «старый слой».

## 3. Концепция нового отображения

Три формата, все на токенах Blanc (radius 16/22/28, `--blanc-surface-strong`, `--blanc-line`,
IBM Plex Sans / Manrope, eyebrow-заголовки):

1. **Right-side panel (основной для форм)** — `Dialog variant="panel"`. Слой прижат к
   правому краю на всю высоту (top/bottom отступы), выезжает справа, список под ним виден —
   **та же механика, что карточка работы**. Закреплённая шапка (`DialogPanelHeader`) → тело
   формы (`DialogBody`, скроллится только оно) → липкий футер действий (`DialogPanelFooter`).
   Одна колонка, ширина по тиру (`default` 520px), мягкие scroll-aware тени вместо линий.
   См. канон в разделе 1. Эталон реализации — `jobs/NewJobDialog`.
2. **Side panel (read-only)** — `FloatingDetailPanel` (`wide`). Детальные карточки, превью,
   история версий. Уже используется — тот же визуальный язык, что и форм-панель.
3. **Page scenario** — самые тяжёлые сценарии (редактор Estimate/Invoice как документ)
   лучше как выделенная страница/маршрут.

Реализуется централизованно: проп `variant` в `DialogContent` (`dialog` | `panel`) —
один источник правды. Редизайн окна-формы = `variant="panel"` (внутренняя разметка тела
не переписывается). Подтверждения остаются `variant="dialog"` + `size`.

## 5. Обоснование классификации

- **R** — у окна ≥2 полей ввода и/или длинная форма / внутренний скролл / зажатый размер
  (`max-w-md/lg`) → восприятие страдает, элементы мелкие. Перевод в `wide`/`full`/panel
  убирает скролл, укрупняет элементы, даёт обзор формы целиком.
- **P** — окно уже использует широкую боковую панель: целевой паттерн, не трогаем.
- **K** — нет формы, только короткий текст + кнопки: компактный центр уместен, UX-проблем нет.

## 6. Рекомендации на будущее (как проектировать новые окна)

1. **Считай поля.** 0 полей + подтверждение → `Dialog size="sm"`. 2+ полей → минимум `wide`.
2. **Не зажимай форму.** Запрещено `max-w-md/lg` для многополевых форм — используй `size`-проп,
   а не ручной `max-w-*` в каждом окне.
3. **Нет внутреннему скроллу.** Если форма не влезает без скролла — это `wide`/`full` или
   2 колонки, или это уже не модалка, а **страница**.
4. **Документ — это страница/панель,** а не модалка (Estimate/Invoice editor, мастера).
5. **Масштаб как на основных экранах** — те же размеры полей/кнопок/типографики (без «мелкого слоя»).
6. **Один источник правды** — размер задаётся `size`-пропом `DialogContent`, дизайн-токенами Blanc;
   не плодить кастомные обёртки.
7. **Единообразие** — однотипные сценарии (все create/edit-формы) — один формат.
