# Спецификация: F011 / Refactor-readiness audit

### Общее описание
Инициатива `refactor-readiness audit` предназначена для подготовки Blanc Contact Center к поэтапному рефакторингу без изменения пользовательского поведения. Цель аудита - зафиксировать фактическое состояние проекта, выявить архитектурный долг, дублирование, legacy-сечения и расхождения между документацией и кодом, а затем оформить это в основу для следующих задач, тест-кейсов и точечных рефактор-слайсов.

### Сценарии поведения

#### Сценарий 1: Инвентаризация фактической структуры проекта
- **Предусловия:** доступен рабочий tree проекта; доступны `docs/`, `src/`, `backend/src/` и `frontend/src/`.
- **Входные данные:** текущий код, документация, список тестов, скрипты сборки и линтинга.
- **Шаги:**
  1. Сопоставить реальную структуру репозитория с описаниями в `requirements.md`, `architecture.md`, `project-spec.md` и `README.md`.
  2. Зафиксировать основные entrypoint'ы, зоны ответственности и слои приложения.
  3. Отметить расхождения между документацией и кодом как отдельные элементы аудита.
- **Ожидаемый результат:** сформирован baseline состояния проекта с указанием актуальных модулей, устаревших описаний и участков, требующих рефакторинга документации.
- **Побочные эффекты:** создается карта расхождений `docs vs code`, которая используется как источник правды для следующих спецификаций.

#### Сценарий 2: Выявление дублирования и смешения ответственности
- **Предусловия:** baseline структуры уже собран.
- **Входные данные:** backend routes/services, frontend services/hooks/components, найденные повторяющиеся паттерны.
- **Шаги:**
  1. Найти повторяющиеся HTTP-клиенты, повторные обращения к одним и тем же API и дублированные UI/logic блоки.
  2. Отметить крупные модули с высокой когнитивной нагрузкой и смешением слоев.
  3. Зафиксировать места, где `db.query` используется напрямую в route-слое вместо выделенного сервиса.
- **Ожидаемый результат:** сформирован список дублирующих и проблемных зон с указанием причин, почему они мешают безопасному рефакторингу.
- **Побочные эффекты:** появляется приоритизированная карта технического долга, пригодная для декомпозиции на независимые задачи.

#### Сценарий 3: Формирование безопасных refactor-slice кандидатов
- **Предусловия:** известны основные риски и дублирование.
- **Входные данные:** список проблемных зон, quality gate результаты, текущие тесты и ограничения по защищенным частям.
- **Шаги:**
  1. Разделить найденные проблемы на безопасные куски, которые можно исправлять без изменения внешнего поведения.
  2. Отделить области, требующие отдельного плана миграции, от тех, которые можно улучшать локально.
  3. Описать порядок рефакторинга по зависимостям и рискам.
- **Ожидаемый результат:** список refactor-slice задач с явными границами, порядком выполнения и зависимостями.
- **Побочные эффекты:** будущие задачи получают четкие ограничения по охвату и не превращаются в "big bang" переписывание.

#### Сценарий 4: Стабилизация quality gate baseline
- **Предусловия:** доступны текущие команды проверки проекта.
- **Входные данные:** `npm test`, `frontend build`, `frontend lint`, отчеты о предупреждениях и ошибках.
- **Шаги:**
  1. Зафиксировать текущее состояние тестов, сборки и линтинга как baseline.
  2. Отдельно выделить уже существующие ошибки, предупреждения и нестабильные тесты.
  3. Отметить разрывы между ожидаемым и реальным качеством проекта.
- **Ожидаемый результат:** создана измеримая отправная точка для контроля качества перед и после рефакторинга.
- **Побочные эффекты:** становится возможной последующая регрессия-проверка на основе одинаковой baseline-метрики.

#### Сценарий 5: Синхронизация проектной документации
- **Предусловия:** аудит структуры, дублирования и качества уже выполнен.
- **Входные данные:** результаты аудита, текущие docs, changelog и project spec.
- **Шаги:**
  1. Обновить `requirements.md` и `architecture.md` так, чтобы они отражали фактическую структуру проекта.
  2. Подготовить перечень новых или уточненных спецификаций и тест-кейсов для приоритетных зон.
  3. Зафиксировать изменения в `changelog.md` и, при необходимости, в `project-spec.md`.
- **Ожидаемый результат:** документация начинает соответствовать коду и может использоваться как опора для следующих изменений.
- **Побочные эффекты:** уменьшается риск того, что новый рефакторинг будет спроектирован на устаревших предположениях.

### Граничные случаи
1. `docs/specs/` пустой или содержит только заглушки -> аудит должен создать первую актуальную спецификацию, а не ссылаться на несуществующие артефакты.
2. Документация сильно расходится с кодом -> приоритет отдается фактическому состоянию репозитория, а расхождение описывается отдельно.
3. В проекте обнаружены legacy-слои и дублирующие entrypoint'ы -> их нельзя объединять в рамках аудита, только классифицировать и описывать.
4. `npm test` или `frontend lint` уже падают до начала изменений -> это фиксируется как baseline, а не трактуется как регрессия от аудита.
5. В рабочем дереве есть артефакты вроде `.bak` или `.DS_Store` -> они отмечаются как техдолг и не должны влиять на оценку поведения приложения.

### Обработка ошибок
1. Если аудит не может сопоставить часть модулей из-за устаревших документов, результатом должен быть список неразрешенных расхождений, а не догадки.
2. Если quality gates не запускаются или дают нестабильный результат, это должно быть явно указано как ограничение baseline.
3. Если найдено несколько конкурирующих реализаций одной функции, аудит должен выбрать канонический путь по фактическому использованию, а не по старому описанию.
4. Если в ходе анализа обнаружена зависимость, которую нельзя безопасно трогать, она должна быть помечена как защищенная зона.

### Взаимодействие компонентов
- `requirements.md` -> определяет существующие продуктовые границы и список фич, которые нельзя ломать.
- `architecture.md` -> фиксирует фактические границы backend, frontend, интеграций и защищенных частей.
- `src/server.js` -> композиционный entrypoint, который связывает legacy-слой и основной backend.
- `backend/src/*` -> основной слой бизнес-логики, маршрутов, сервисов и БД-доступа.
- `frontend/src/*` -> основной UI-слой, где видны дубли запросов, хуков и компонентов.
- `docs/changelog.md` -> журнал аудита и источник для следующих итераций.
- `docs/specs/` и `docs/test-cases/` -> конечные артефакты для refactor-slice задач.

### API-контракты
Для этой инициативы новых runtime API-контрактов нет. Аудит не добавляет endpoints и не меняет поведение существующих API; он только фиксирует их состояние и границы для будущего рефакторинга.

### Артефакты аудита
- `docs/requirements.md` - обновленные требования с привязкой к фактическому коду.
- `docs/architecture.md` - актуализированная архитектурная карта.
- `docs/specs/refactor-readiness-audit.md` - спецификация инициативы.
- `docs/test-cases/refactor-readiness-audit.md` - тест-кейсы для quality gate и refactor-slice сценариев.
- `docs/tasks.md` - декомпозиция на атомарные задачи.
- `docs/changelog.md` - запись о проведенном аудите.

### Критерий готовности
Инициатива считается готовой, когда есть подтвержденный baseline качества, карта расхождений docs vs code, список дублирования и технического долга, а также набор безопасных refactor-slice направлений с понятными границами и приоритетами.
